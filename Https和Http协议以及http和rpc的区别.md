# RPC和Feign或者是OpenFeign的区别？难道不算RPC框架？？这个都得看定义！

## ✅ 一、什么才是真正意义上的 RPC 框架？

### 🌟【定义】

> #### **RPC（Remote Procedure Call）框架** 是一种允许调用远程主机上的函数，就像调用本地函数一样的技术。它**屏蔽了网络通信的细节**，追求“**本地调用体验**”。
>
> ### 换句话说——用户在OpenFeign或者是 Feign中 还是需要理解 Http的一些东西的，比如  需要用户理解 URL、Header、Method 等。没有屏蔽网络通信的细节

也就是说：

| 特征       | 真·RPC 框架必须具备                               |
| ---------- | ------------------------------------------------- |
| 💻 调用方式 | `HelloService.hello("xxx")`，像本地方法一样       |
| 🔌 底层传输 | 通常是 TCP（gRPC 也是 HTTP/2，但强约定结构）      |
| 📦 通信结构 | 请求/响应是自定义的二进制格式或压缩格式（高性能） |
| 🧠 框架核心 | 动态代理 + 编解码器 + 注册中心 + 通信层           |

## 🚫 二、为什么 Feign/OpenFeign 不算真·RPC 框架？

### 🔍 本质分析：

Feign 是什么？OpenFeign 是什么？

> 它们是基于 **HTTP 协议** + **注解驱动的 REST 客户端**。

虽然调用方式看起来像 RPC（也是接口调用），但其**本质依赖 HTTP 协议**，底层就是用 `RestTemplate` 或 `HttpClient` 发请求。

| 特征           | Feign/OpenFeign                                      |
| -------------- | ---------------------------------------------------- |
| 🔌 协议         | HTTP/1.1                                             |
| 📤 报文格式     | 明文 JSON / XML                                      |
| 📚 数据传输语义 | 基于 REST/HTTP 动作（GET/POST）                      |
| 🚪 服务发现     | 一般搭配 Spring Cloud + Eureka/Consul                |
| ❌ 网络抽象     | 无法脱离 HTTP 协议，暴露 HTTP 语义（header, status） |

因此，它不是**屏蔽网络细节**、打造“像调用本地函数一样”的抽象，而是：

> ⛔️ 一种 **HTTP 请求发起工具 + 注解封装 + 序列化自动绑定**。

## ✅ 三、Dubbo / Motan / gRPC 和 Feign 的本质差异

| 项目     | Dubbo / Motan / gRPC                      | Feign / OpenFeign                   |
| -------- | ----------------------------------------- | ----------------------------------- |
| 协议     | TCP / HTTP2 / 自定义协议                  | HTTP/1.1                            |
| 编码方式 | Protobuf / Hessian / 自定义二进制         | JSON                                |
| 性能     | 高吞吐、低延迟                            | 较低，解析成本大                    |
| 通信抽象 | 自定义 Request/Response，甚至自定义 Codec | 直接暴露 HTTP 请求结构              |
| 网络细节 | 封装到底层不可见                          | 需要用户理解 URL、Header、Method 等 |
| 定位     | 真正的 RPC 框架                           | HTTP 调用工具封装器                 |

## ✅ 四、打个比方你就懂了：

- **Dubbo / gRPC / Motan：**

  > 像你打电话，打通了之后你说什么就是“函数调用”，完全不用知道电话协议细节。

- **Feign / OpenFeign：**

  > 像你发一封邮件（HTTP 请求），你得知道邮件头要怎么写，地址是什么格式，邮件体怎么填，邮箱格式有没有问题……

**一句话总结：**

> Feign 本质上是个“注解 + HTTP 工具框架”，不是网络协议抽象层，不配称作真·RPC 框架。

------

## ❓是不是说用 HTTP 就不能是 RPC？

不是。关键看有没有“把 HTTP 抽象掉”，比如：

- gRPC 用的是 HTTP/2，但它封装成了 `stub.call()`，你不会看到任何 HTTP。
- RESTTemplate、Feign、Retrofit 都暴露了 HTTP 概念，所以不算 RPC 框架。





# TCP和HTTP/1.1、HTTP/2

## ✅ 一句话告诉你它们的层级（必须牢记）

| 协议名               | 属于哪一层？   | 你要关心的关键区别是？                                       |
| -------------------- | -------------- | ------------------------------------------------------------ |
| **TCP**              | **传输层协议** | 保证数据可靠送达，但不管“数据内容”                           |
| **HTTP/1.1、HTTP/2** | **应用层协议** | 具体定义了：请求怎么发、响应怎么回、header长啥样、body怎么包 |

换句话说：

> TCP 是一条“水管”，HTTP 是在水管中送水的“水桶样式”。

## ✅ 一、TCP 是什么鬼？

**Transmission Control Protocol**

它是传输层的协议，重点特性如下：

- **面向连接**（需要三次握手建立连接）
- **可靠传输**（自动重传、流控、顺序保证）
- **不关心内容是什么**（是 JSON 还是 Protobuf 它不在乎）

💡 **TCP 不定义你传什么内容，只负责稳定送过去！**

## ✅ 二、HTTP 是什么鬼？

HTTP 是“**基于 TCP 的** 应用层协议”，也就是说：

- HTTP **必须跑在 TCP 上**。
- 它定义的是请求和响应的结构，比如你熟悉的：

```
GET /hello HTTP/1.1
Host: www.example.com
User-Agent: xxx
Content-Type: application/json

{"name":"郑少"}
```

这种结构，就是 HTTP 协议定的格式。

## ✅ 三、HTTP/1.1 vs HTTP/2 差异

### HTTP/1.1（老一代协议）

- 每次请求只能发一个内容（阻塞）
- 多个请求复用一个连接要排队（队头阻塞）
- 全是明文（Header 冗长）
- 无多路复用、无压缩

👉 就像一条马路，一次只能通一辆车。

------

### HTTP/2（新一代协议）

- **多路复用**：多个请求能并行共享同一个 TCP 连接
- **Header 压缩**：体积小，减少带宽
- **二进制帧**传输：比文本结构更高效

👉 就像高速公路，多个车道，能并发处理多个请求。

## ✅ 四、那“自定义协议”又是啥几把？

真·RPC 框架（Dubbo、Motan）很多时候根本不用 HTTP 协议！

> 它们直接自己定义通信格式、字段、结构，甚至自己设计“魔数、长度、序列化方式”等。

比如 Dubbo 的二进制协议一段数据长这样（举个例子）：

```
[魔数][版本][请求id][请求体长度][请求体内容]
```

你根本看不到什么 GET、POST、Header 这些东西，它直接就是**裸 TCP + 自定义结构 + 高性能编解码**。

------

## ✅ 五、为什么这事儿对你写 RPC 框架重要？

因为你写的框架，是要做这个决策的：

| 选项                            | 影响点                                      |
| ------------------------------- | ------------------------------------------- |
| 用 HTTP                         | 简单、容易调试（curl 能看），但性能低，包大 |
| 用 TCP + 自定义协议             | 复杂一点，但性能高、结构可控、头部更紧凑    |
| 用 HTTP/2 + Protobuf（如 gRPC） | 性能折中，语义清晰，官方支持强，但依赖多    |

所以你现在学的 `guide-rpc-framework` 是基于 Netty + TCP + 自定义协议的，这是走性能路线的正经 RPC 框架写法。

## ✅ 总结一句话给你背：

> TCP 是“水管”，HTTP 是“桶的样子”，HTTP/1.1 桶装水，HTTP/2 是高压喷头，RPC 框架可以选择不用桶自己造容器（自定义协议）。



# HTTP 为什么被说“很重”？

## ✅ 一、HTTP 为什么被说“很重”？真相全揭露！

“重”这个词，在协议领域里的意思其实是三个维度的“成本叠加”：

| 类别           | 解释                                             | 影响                     |
| -------------- | ------------------------------------------------ | ------------------------ |
| 📦 报文体积大   | header太多、格式冗余、用明文                     | 每次传数据浪费带宽和时间 |
| 🧠 解析麻烦     | 是文本结构（需要字符串解析）                     | 解析慢，CPU占用高        |
| 🧱 协议语义复杂 | 有很多不必要的语义（状态码、cookie、header字段） | 你不一定用得上，但它都在 |

我们来逐条掰开看。

## 📦 1. 报文头太长 = 带宽浪费 + 性能浪费

举个栗子：

```
GET /api/hello HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0 (Macintosh; Intel Mac OS X)
Accept: */*
Accept-Encoding: gzip, deflate, br
Connection: keep-alive
Authorization: Bearer xxxx
```

👆这部分 Header 就有 **上百字节甚至几 KB**，而你真正要传的内容，可能只有一个 JSON：

```
{"id": 1}
```

### 🔥 总结一句话：

> **为了传10字节数据，你可能要“背着几百字节的行李”走一趟。**

这对高频通信、低延迟系统（比如游戏、金融）是灾难。

------

## 🧠 2. 文本格式 = 字符串解析地狱

#### HTTP 是明文协议，所有东西都要用字符串处理：

- 请求行要拆 `GET /url HTTP/1.1`
- 每一行 Header 都要 `split(": ")`
- 如果有 body，还得按 `Content-Length` 手动截取
- JSON 还要二次反序列化

### 🔥 问题在于：

> **文本结构不仅体积大，而且解析起来极慢！**（对比 Protobuf 二进制结构的性能差了一个量级）

## 🧱 3. 协议功能太多，用不上但必须带

HTTP 是为了浏览器而生的，它默认支持很多东西你可能根本不需要：

- Cookie
- 302 重定向
- Header negotiation
- Cache control
- Accept-Language / Accept-Encoding 等内容协商字段
- 状态码 1xx~5xx，几乎没人用全

而你做 RPC 调用，只想做一件事：

```
HelloService.sayHi("郑少") --> 返回字符串
```

结果你却不得不“穿西装戴领带”走 HTTP。

## 🔍 那自定义协议真的更轻吗？是的，因为你可以只传你想传的

举个例子，假设你自定义协议只要这些字段：

```
[magic:2B][version:1B][type:1B][requestId:8B][bodyLength:4B][body:N]
```

- 全是定长字段 + 二进制结构
- 解析时直接按 offset + length 取数据，速度贼快
- 没有字符串比对、没有多余字段、不需要任何 header negotiation

这就像什么？

> **HTTP 是大超市的购物车（啥都有但沉），而自定义协议是你量身定做的快递袋（只装你要的）**

## 📌 小结：为什么说 HTTP 很重

| 点       | HTTP 的问题                    | 自定义协议的优势             |
| -------- | ------------------------------ | ---------------------------- |
| Header   | 结构庞大，冗余字段多           | 可以完全精简                 |
| 数据格式 | 文本结构，需字符串解析         | 二进制定长，直接偏移读取     |
| 语义     | 协议逻辑复杂，不可裁剪         | 全部自己定                   |
| 带宽     | 明文传输，无压缩（除非你手动） | 通常压缩 + 紧凑结构          |
| 性能     | CPU解析成本高                  | 快速反序列化，零拷贝更容易做 |

## 🚫 最后打破一个误区：

> ❌ “HTTP 很成熟很流行，那它一定适合所有场景”——错！

它适合人机交互、调试友好、不高频场景。但在高性能分布式场景（RPC、游戏、金融）下，它就是个🐘——**能跑，但太重了，性能堪忧。**





# HTTP的明文体现在哪里呢？

## ✅ 一、什么叫做“明文协议”？HTTP 的明文到底是哪里明？

### 🧠 定义：

> **明文协议** 是指其通信内容在网络上传输时是“**直接以字符串**（纯文本）表示”，而不是经过压缩、加密、编码或二进制处理。

换句话说：

- 你可以用 `telnet` 或 `nc` 连接 HTTP 服务，直接“肉眼”看懂通信内容
- 抓包工具（如 Wireshark）看到的是 **人能直接读懂的东西**
- 所有结构：请求行、header、body 都是字符串拼接

## 👀 举个现实中的明文 HTTP 请求（不带 JSON）

```
POST /api/hello HTTP/1.1
Host: www.example.com
Content-Type: application/json
Content-Length: 18

{"name":"郑少"}
```

你看这整个结构：

| 部位   | 内容                       | 都是啥                    |
| ------ | -------------------------- | ------------------------- |
| 请求行 | `POST /api/hello HTTP/1.1` | 完整一行字符串            |
| 请求头 | `Content-Type: xxx`        | 每一行都需要按 `: ` 分割  |
| 请求体 | `{"name":"郑少"}`          | 是 JSON，是结构化的字符串 |



> 这个格式叫做“**纯文本拼接的结构协议**”。

## ✅ 二、明文协议 vs 二进制协议，对比你就懂了

| 协议类型       | 特征                             | 举例                       | 优点                 | 缺点               |
| -------------- | -------------------------------- | -------------------------- | -------------------- | ------------------ |
| **明文协议**   | 全是字符串，结构靠空格或换行分隔 | HTTP、SMTP、FTP            | 人类友好，调试方便   | **体积大，解析慢** |
| **二进制协议** | 定长字段，结构靠 offset + length | Dubbo 协议、Protobuf、gRPC | 紧凑、高效、易于解析 | 不方便人工查看     |

比如 Dubbo 协议的一段数据是这样的：

```
0xdabb (magic)
0x02 (version)
0x01 (flag)
0x0000000000000013 (requestId)
0x0000001A (dataLength)
[序列化后的 data]
```

你看不懂，但机器只要按字段偏移量就能读出来，非常快。

## ✅ 三、为什么“明文协议 = 字符串处理地狱”？

我们以你调用一次 HTTP 接口为例，来看看 HTTP 服务端需要干多少破事：

### 服务端解析 HTTP 请求的步骤（不含业务逻辑）

1. **读取 TCP 字节流**（从 socket 读出来）
2. **转换为字符串**
3. **按行分隔字符串**，提取出请求行、header、空行、body
4. **请求行再切割成 3 个字段**：`方法 路径 协议版本`
5. **header 每一行再 `split(": ")` 分为 key 和 value**
6. **按 Content-Length 计算 body 长度**
7. **再解析 body 内容**（比如是 JSON，就得再反序列化）

🔥 以上全部操作 = 全部靠 **字符串切割 + 正则 + 手动偏移处理**。

相比之下，**二进制协议**会这么干：

```
magic = bytes.readShort();        // 2字节
version = bytes.readByte();       // 1字节
requestId = bytes.readLong();     // 8字节
length = bytes.readInt();         // 4字节
body = bytes.readBytes(length);   // 一次拿完
```

一行搞定，不涉及任何字符串操作，速度快如闪电。

------

## ✅ 四、你说“不是都传 JSON 吗？”的问题拆解

你说得没错，很多 HTTP 请求的 body 是 JSON 没错，比如：

```
{"username":"郑少","password":"123456"}
```

但是请注意：

1. **HTTP 协议本身不止是 JSON**，它还要你加一堆 **header**
2. 即使你传 JSON，你也得 **先过完前面的所有 HTTP 字符串处理逻辑**，再去处理 JSON
3. 而 JSON 本身也比 Protobuf、Avro 这些二进制结构“更重更大更难解析”

所以：

> 💥 **HTTP + JSON 是两层明文协议叠加 = 明文地狱套娃结构！**

## ✅ 五、总结：HTTP 明文到底“明”在哪里，麻烦在哪里？

| 问题点        | 说明                                         | 成本                           |
| ------------- | -------------------------------------------- | ------------------------------ |
| 明文结构      | 整个协议都靠换行、冒号分割，没有 offset 概念 | 内存复制多、解析速度慢         |
| 多行拼接      | 每一行都要单独处理，header多则更慢           | header 越多越慢                |
| 依赖文本格式  | 无法提前知道包长、字段偏移                   | 不能一次性预读、无法 zero-copy |
| 解析器复杂    | HTTP Server 都得实现 HTTP parser（状态机）   | 编码复杂、Bug 多、安全风险高   |
| JSON 本身也重 | 序列化慢、体积大                             | CPU 占用更高、网络传输慢       |

## 🧠 你要记住的结论是：

> **HTTP 是明文协议**，因为它的结构完全基于字符串拼接；
>
> 你想快速传数据，就要“越少字符串、越少字段、越少结构越好”，而 HTTP 恰好反其道而行之。





# HTTPS是明文传输的吗？？和HTTP的区别在那里？

* #### https内容是明文结构，不过传输是加密的。

## ✅ 一句话回答你：

> **HTTPS = 加密传输的 HTTP，内容原本是明文协议，但传输过程中是加密的，抓包看不到。**

所以：

- ✅ **HTTPS ≠ 明文传输** ❌（传输是密文）
- ✅ **HTTPS 是基于明文协议的加密外壳** ✔️（结构仍是 HTTP，只是被 TLS 包起来了）

## ✅ 二、图解区别：HTTP vs HTTPS

| 特性             | HTTP                              | HTTPS                              |
| ---------------- | --------------------------------- | ---------------------------------- |
| 协议类型         | 明文协议                          | 明文协议 + TLS加密外壳             |
| 抓包能看到什么？ | 所有内容都能看见（Header + Body） | 都是加密的，看不到任何有用内容     |
| 安全性           | 无任何安全性                      | 可防止中间人、劫持、窃听           |
| 内容格式         | 字符串拼接（GET /xx HTTP/1.1）    | 一模一样，只是传输加密             |
| 明文处理         | 本质是明文协议                    | 本质是明文协议，但**传输层已加密** |

### 📦 拿抓包工具一对比你就明白了：

#### 📨 HTTP 抓包：

```
GET /api/user HTTP/1.1
Host: www.example.com
Authorization: Bearer xxx
Content-Type: application/json

{"id":1}
```

#### 🔐 HTTPS 抓包（未做中间人攻击）：

```
Encrypted Application Data (TLSv1.3)...
[无法读取内容]
```

你只看到“加密的数据流”，看不到请求行、header、body。

## ✅ 三、HTTPS 是怎么加密 HTTP 的？

我们说“HTTPS 是 HTTP over TLS（或 SSL）”，这代表：

- 建立连接前，先走一套 **TLS 握手流程**（公钥交换、密钥协商）
- 握手成功后，才开始用这个对称密钥**加密 HTTP 的内容**
- 实际 HTTP 报文格式**并没有变**，只是**都被 TLS 封装并加密**

🔑 注意重点：

> **TLS 是加密管道，HTTP 是在管道里的明文结构**

所以说：

- HTTP 的“明文结构”没有变
- HTTPS 只是让它**在路上传输时被加密了**

## ✅ 四、HTTPS 还是不是“明文协议”？

这个问题你要看角度：

| 角度            | 是否是明文                                       |
| --------------- | ------------------------------------------------ |
| 📡 网络传输层    | ❌ 不是（因为 TLS 加密）                          |
| 📄 协议格式层    | ✅ 是（因为它用的仍是 HTTP 的明文结构）           |
| 👨‍💻 服务端内存里 | ✅ 是（你收到包之后必须解密成明文 HTTP 才能处理） |

## ✅ 五、HTTPS 和自定义 RPC 协议差距在哪？

| 点       | HTTPS                   | 自定义协议（比如 Dubbo）       |
| -------- | ----------------------- | ------------------------------ |
| 结构     | HTTP 明文结构（头很重） | 自定义结构（定长字段，紧凑）   |
| 安全性   | TLS 加密保障            | 你可以选加密也可以不加（灵活） |
| 报文体积 | header 多、结构复杂     | 极简字段，节省传输成本         |
| 解析方式 | 字符串解析 + JSON       | 二进制 + 直接偏移读取          |
| 调试难度 | 浏览器友好，易抓包      | 不可读，但更高效               |

## ✅ 总结结论（建议背下来）：

> **HTTPS 是传输加密了的 HTTP，虽然传输过程不是明文，但协议本质还是明文结构。**
>
> 所以它比 HTTP 安全，但依然“重”；它比自定义协议安全，但“性能成本高”。





# HTTPS为什么需要证书呢？（因为浏览器不相信你）

* #### 🧩 先明确结论：你之所以需要证书，是因为浏览器不信你！

## 🧩 先明确结论：你之所以需要证书，是因为浏览器不信你！

HTTPS 的核心功能是：**让客户端相信“我真的是郑少的网站”，并且我们的通信不被中间人窃听**。

这靠三件事实现：

| 核心问题               | 解决方式                                 |
| ---------------------- | ---------------------------------------- |
| 你是谁？               | 用“证书”证明                             |
| 你说的是真话吗？       | 用“CA 签名”认证                          |
| 咱们说的别人能听到吗？ | 用“对称密钥”加密（但它怎么来的？——协商） |

## ✅ 阶段一：为什么你的网站必须“申请证书”？

因为——

> **HTTPS 本质是“我宣称我是 example.com，客户端凭证书判断信不信”**

但是问题是：

- 你自己生成一个证书完全可以，但客户端（比如浏览器）根本不信你
- 浏览器只相信“**根证书在它自己信任列表里**”的那一批机构（CA）

所以你必须：

- 从 **被主流浏览器信任的 CA（证书颁发机构）** 那里，花钱（或用 Let's Encrypt 免费）申请一个证书
- 它会生成 `.crt + .key` 文件，供你的服务端使用（比如 Nginx）

## ✅ 阶段二：TLS 握手流程（详细版）🧠（这个是 TLS 1.2版本的 基于  RSA旧版的）

我们来详细看下**客户端和服务端如何协商加密通信**：

```
客户端（浏览器）           服务端（你的网站）
      |                        |
1.    | --- ClientHello -----> |  🔼 说：我支持 TLS1.3，用这些加密算法
2.    |                        |       + 我想连的是 www.zhengshao.com
3.    | <--- ServerHello ----- |  🔽 说：我选 TLS1.3，选这种算法
4.    | <--- 发送证书 -------- |  🔽 给你我网站的证书（带公钥）
5.    |                        |     
6.    | --- 验证证书 ----------|  🔍 客户端用内置“根证书”检查：
                                签发者是谁？域名匹配吗？有效期内吗？
7.    | --- 生成对称密钥 ----> |  🔐 浏览器生成随机密钥，用服务端“公钥”加密后发送
8.    |                        |  🔑 服务端用私钥解密，拿到对称密钥
9.    |                        |
10.   | <===> 开始加密通信 <==> | 📡 双方开始用协商出来的密钥，进行对称加密通信
```

## ✅ 重点步骤详解

### 🔑 公钥和私钥用在哪？

- 你的网站证书中包含：**公钥**
- 服务端保存的 `.key` 文件是：**私钥**

客户端拿到你的证书 → 用公钥加密一个随机对称密钥 → 服务端用私钥解密 → 双方就共享这个密钥 → 后续通信都走对称加密（AES、ChaCha20）

## 🧠 那“密钥协商”用的是啥算法？

- TLS1.2 以前用 RSA 握手：直接用公钥加密对称密钥
- TLS1.3 推荐 ECDHE（椭圆曲线 Diffie-Hellman）：更安全、更快，支持前向安全

所以：

> **整个 TLS 只用一次非对称加密（公钥 + 私钥），后续都用对称加密（速度快）**

## ✅ 阶段三：证书的验证逻辑

### 浏览器是怎么验证你的证书可信的？

它内置了 100 多个“根证书”，比如：

- DigiCert
- Let's Encrypt
- GlobalSign

你的网站证书会带有“签名链”，从你的证书 → 中间证书 → 根证书

浏览器会一层层检查这个链：

- 证书是从可信 CA 来的吗？
- 域名是否匹配？
- 是否过期？
- 签名是否有效？

如果某一层失败，就报错：`证书无效`、`无法建立安全连接`。

## ✅ 阶段四：为什么一定要 HTTPS？HTTP 有什么风险？

| 风险           | 说明                                             |
| -------------- | ------------------------------------------------ |
| 🕵️ 中间人攻击   | 比如公共 WiFi 拦截你和服务端之间的包，篡改       |
| 🧅 窃听敏感信息 | 抓包可看到用户名、密码、Cookie                   |
| 🔀 数据篡改     | 中间人可以修改请求体或响应体（比如替换 JS 文件） |
| ❌ 身份伪造     | 如果没证书，你连的是谁完全说不清楚               |

所以：

> **有了 HTTPS + 合法证书，才能防止劫持、监听、身份伪造**，保证通信的“保密性 + 完整性 + 真实性”。

------

## ✅ 小结整理（送你一张核心图）：

```
          浏览器       🔒            网站服务器
         ┌──────┐   TLS握手    ┌──────────────────────┐
         │根证书库│────────────→│ 证书（包含公钥）+ 私钥 │
         └──────┘               └──────────────────────┘
             ▲                         ↓
     验证证书是否合法         私钥用于解密对称密钥
             ↓                         ↓
         通信加密开始 ←─────→ 对称密钥 ←──── 浏览器生成
```

## 🧠 总结记忆口诀：

> 证书是“你是谁”的身份证，
>  公钥是“接收信”的锁，
>  私钥是“看信”的钥匙，
>  对称密钥是“咱俩的密语”，
>  TLS 就是建立密语的流程！





# TLS 1.2 以及 TLS 1.3的区别

## ✅ 一、TLS1.2 中的两种握手方式（都有）

| 模式                   | 对称密钥的来源                           | 说明                                  |
| ---------------------- | ---------------------------------------- | ------------------------------------- |
| **RSA 握手（老旧）**   | 由客户端随机生成，对称密钥是“客户端造的” | 然后用服务端公钥加密传过去            |
| **ECDHE 握手（现代）** | 双方协商，各自生成密钥对计算得出         | 每次握手密钥不同，支持前向安全（PFS） |

👉 很多 TLS1.2 的实际实现（比如早期 Nginx）默认用 RSA，所以网上有一堆人讲的是旧逻辑。

## ✅ 二、TLS1.3 中握手方式（只剩一种）

> **强制使用 ECDHE（Ephemeral Diffie-Hellman）+ AEAD 加密**，RSA 握手彻底废除！

### 所以 TLS1.3 的握手关键点是：

```
双方生成临时密钥对 → 交换公钥 → 各自用对方公钥 + 自己私钥算出共享密钥（session key）
```

这个过程是真·协商出来的对称密钥，**不是任何一方“生成后发送”的**。



# TLS1.2 的 RSA 握手流程

## 🧠 重点：你看到“客户端生成密钥然后用公钥加密发过去”的逻辑是 TLS1.2 的 RSA 握手！

TLS1.2 的 RSA 模式长这样：

1. 服务端发证书（里面是 RSA 公钥）
2. 客户端验证证书
3. 客户端随机生成 session key（对称密钥）
4. 客户端用服务端公钥加密 session key 发过去
5. 服务端用私钥解密，拿到 session key
6. 后续使用 session key 进行对称加密

这个流程中：

✅ 是客户端自己造的密钥
 ❌ 没有协商过程
 ❌ 没有前向安全性（私钥泄露 = 过去通信全暴露）

* #### 一个用了一个半来回，所以约等于两个的。



#  TLS1.3 握手流程 —— 唯一的 ECDHE 模式（详细 + 不跳步）

### 🎯 目标：

建立一条安全通信信道，让客户端和服务端都拥有一个**一样的对称密钥**（session key），但中间人哪怕抓包也猜不到。

------

## 🔐 握手整体分 3 个阶段：

| 阶段                   | 内容                                                         | 目的                       |
| ---------------------- | ------------------------------------------------------------ | -------------------------- |
| ① 客户端发 ClientHello | 提供支持的加密算法、curve、协议版本 + **发送自己的 ECDHE 公钥** | 启动握手 + 准备协商密钥    |
| ② 服务端发 ServerHello | 回复选定算法 + **发送服务端的 ECDHE 公钥** + 证书（含签名）  | 确定加密参数 + 证明身份    |
| ③ 双方计算共享密钥     | 各自用对方公钥 + 自己私钥计算共享 secret                     | 得到对称密钥，开始加密通信 |

## 🧠 详细步骤（TLS1.3 的标准过程）

### ✅ 第 1 步：ClientHello（发起方）

客户端发送的内容包含：

| 字段                                 | 含义                                    |
| ------------------------------------ | --------------------------------------- |
| 支持的协议版本                       | 比如 TLS1.3                             |
| 支持的加密算法                       | 比如 AES_128_GCM、ChaCha20              |
| 支持的密钥交换算法                   | 比如 ECDHE using Curve25519 / secp256r1 |
| 客户端随机数（client_random）        | 用于后续计算共享密钥                    |
| ✅ 客户端的 ECDHE 公钥（X25519 公钥） | 🔥这是关键                               |

这一步客户端就已经主动生成了一对密钥（私钥 + 公钥），把公钥发出去了。

------

### ✅ 第 2 步：ServerHello（回应方）

服务端收到 ClientHello 后：

1. 验证支持的协议版本和算法，选一个双方都支持的
2. 生成自己的 ECDHE 密钥对（私钥 b、公钥 B）
3. 回应内容包括：

| 字段                                         | 含义                       |
| -------------------------------------------- | -------------------------- |
| 选定的算法版本                               | 比如 TLS1.3, AES128-GCM    |
| 选定的 ECDHE 参数                            | 比如 curve25519            |
| ✅ 服务端的 ECDHE 公钥                        | 🔥这是关键                  |
| ✅ 证书（包含签名 + 公钥）                    | 用于证明身份               |
| ✅ 证书签名（签了 ClientHello + ServerHello） | 防止中间人伪造 ServerHello |

> 🔐 这个签名非常关键，是防止中间人攻击的核心，证明“我真的是 example.com 的服务器”。

------

### ✅ 第 3 步：双方各自计算共享密钥

这时候客户端和服务端各自都有了：

| 谁     | 拥有的东西                             |
| ------ | -------------------------------------- |
| 客户端 | 私钥 a、自身公钥 A、收到的服务端公钥 B |
| 服务端 | 私钥 b、自身公钥 B、收到的客户端公钥 A |

然后各自执行：

```
共享密钥 = 公钥^私钥（在椭圆曲线上）
K_client = (B)^a
K_server = (A)^b
```

由于数学算法保证结果相同，因此：

✅ `K_client == K_server`

这个 `K` 就是双方的共享 secret，接下来再用它通过 HKDF 派生出：

- 对称加密密钥（session key）
- 消息验证码（MAC）
- Finished 验证字段

## ✅ 第 4 步：双方切换为加密通信

1. 双方发出 `Finished` 报文：这是第一个**用对称密钥加密**的消息，用来确认之前的 handshake 没有被篡改
2. 双方计算并校验 HMAC
3. 成功后，握手完成





# 为什么 TLS1.3 改成客户端先发？

## ✅ 结论先行：为什么 TLS1.3 改成客户端先发？

> 因为：
>
> 1. 🚀 性能角度：客户端先发 ECDHE 公钥 → **缩短握手轮次**，实现 "1-RTT"（一次来回）完成握手
> 2. 🔐 安全角度：握手早早就把加密基础打好，服务端无法替换加密参数
> 3. 💡 模型转变：从“我服务端给你密钥解锁” → “咱俩一起造密钥”

## 🧨 回顾：TLS1.2 的 RSA 握手流程为什么是“服务端先发”？

### 简化版逻辑如下：

1. 客户端：发 `ClientHello`，说“我要连你，我支持 TLS1.2 和这些加密算法”
2. **服务端先发证书 + 公钥（RSA）**
3. 客户端生成对称密钥 session key，用服务端公钥加密发回
4. 服务端用私钥解密，握手结束

### 👉 所以顺序上：必须等服务端先发公钥，客户端才能“发密钥”。

------

## ⚠️ 这个顺序有几个明显的问题（TLS1.2 的弱点）：

| 问题                                 | 说明                               |
| ------------------------------------ | ---------------------------------- |
| ❌ 服务端公钥先发，容易遭遇中间人伪造 | 没有任何校验之前你就开始“信”服务端 |
| ❌ 客户端不能提前准备密钥交换材料     | 无法实现 0-RTT/1-RTT               |
| ❌ 握手至少要 2 个来回（RTT）         | 慢，尤其是在移动网络下明显卡顿     |
| ❌ 无前向安全性                       | 私钥一旦泄露，历史数据全挂         |

## ✅ TLS1.3 为什么要改成“客户端先发”？【核心转变】

### 🔥 原则一：握手不能慢 → 必须提前发密钥交换参数（ECDHE 公钥）

- 客户端主动发出自己的 ECDHE 公钥
- 服务端回发自己的公钥 + 签名 + 证书（一次性完成身份认证 +算法协商）
- 握手完成后直接切入加密通信，整个过程只需要**1 个 RTT**（网络来回）

👉 **这叫做 TLS1.3 的 1-RTT 设计核心。**

------

### 🔥 原则二：握手必须安全 → 服务端不能中途更改密钥交换算法

TLS1.2 的漏洞在于：

- 服务端可以伪装支持较弱算法（被称为“降级攻击”）
- 客户端没法控制对方到底用什么

TLS1.3 让客户端先表态算法 + 先送出 ECDHE 公钥，是为了保证：

> 服务端在签名时必须绑定 ClientHello（包含公钥和加密算法）
>  这样如果有人在中间篡改握手内容，签名校验会直接失败。

✅ 这保证了握手过程不能被篡改，抵御中间人攻击。

------

### 🔥 原则三：握手必须支持 0-RTT（可选）

TLS1.3 允许开启 **0-RTT 模式**：客户端可以在第一次数据包里就发业务数据！

前提就是：

- 客户端必须已经提前准备好了密钥交换内容
- 所以必须让客户端成为握手主导方

## ✅ 小结对比图（为啥 TLS1.3 要反转发起顺序）

| 对比点             | TLS1.2（RSA）                | TLS1.3（ECDHE）                |
| ------------------ | ---------------------------- | ------------------------------ |
| 握手主导方         | 服务端先发公钥               | 客户端先发公钥                 |
| 轮次               | 至少 2 个 RTT                | 只需 1 个 RTT                  |
| 安全性             | 无前向安全，容易被中间人篡改 | 默认前向安全，绑定签名不可篡改 |
| 是否支持 0-RTT     | ❌ 不支持                     | ✅ 支持（选配）                 |
| 谁生成 session key | 客户端造好发过去             | 双方协商计算                   |

## 🧠 你可以总结为一句话：

> TLS1.3 设计者把握握手主动权交给了客户端，不仅是为了更快，还为了更安全。

它的底层逻辑是：

> “我（客户端）必须一开始就把我的身份和加密方案说清楚，这样你（服务端）就别想中途换条件搞我了。”
