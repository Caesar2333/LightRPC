# 序列化的定义是什么呢？

### 📌 序列化（Serialization）：

> #### 是指将 **Java 对象 → 转成字节数组** 的过程，方便网络传输或存储。

比如你有一个对象：

```
User u = new User("张三", 20);
```

你需要把这个对象转成 byte[] 发给对方，那么你必须做“序列化”：

```
byte[] bytes = serialize(u);
```

反过来，对方收到 byte[]，要变成 Java 对象，就要“反序列化”：

```
User u = deserialize(bytes);
```

## 序列化是有很多种类的。

* #### 将`Java`对象转换成json字符串，再将json字符串编码成字节数组，最后作为消息的一员传输出去。——文本序列化。Jackson 是“结构+文本”型序列化，调试友好。

  * Jackson 是文本型 JSON 序列化器，适合通用、跨语言结构

  * #### 为什么叫做作为消息的一员？因为我们序列化的目标只有消息体内容，换句话说，只有其中的java对象转换成字节，其他的header或者是其余的东西没有必要序列化。序列化的定义本来就是将`将Java对象转换成字节的`

  * #### 我们将`Java`对象转换成json字符串，再通过编码抓换成字节的方式，称作为文本序列化。因为最后转换成的时字符串。其内容是可以清晰地被解码之后看到的，方便调试。这也是为什么其能够跨平台的原因。因为其不用使用json反序列化，将json字符串成java对象就可以看到其内容的。是非常方便的。

* #### 将`Java`对象直接转换成二进制数组的，在没有抓换成反序列化成`Java`对象之前是不可以看到其内容的。Kryo/Protobuf 是“结构+二进制”型序列化，性能最优但不可直接查看，必须配合反序列化才能解读。

  * #### Kryo/Protobuf 是压缩型二进制序列化器，适合高性能传输



# 序列化有多少种呢？？

* #### Jackson 是文本型 JSON 序列化器，适合通用、跨语言结构

* #### Kryo/Protobuf 是压缩型二进制序列化器，适合高性能传输

## ✅ 一、序列化分类：不是说“序列化”就完了，它本身有多种类型

| 序列化类型       | 描述                                   | 示例                             | 可读性 | 体积   | 通用性            |
| ---------------- | -------------------------------------- | -------------------------------- | ------ | ------ | ----------------- |
| 文本序列化       | 把对象变成结构化字符串                 | JSON（Jackson）、XML、YAML       | ✅ 强   | ❌ 大   | ✅ 高              |
| 二进制序列化     | 把对象字段变成原始字节、编号、压缩形式 | Protobuf、Kryo、Hessian、Avro    | ❌ 无   | ✅ 小   | Protobuf 可跨语言 |
| 编码型压缩序列化 | 特殊算法压缩对象（如LZ4+Schema）       | Fastjson+Gzip、Snappy+自定义结构 | ❌ 弱   | ✅ 极小 | 视情况            |

🧠 **扩展概念**：序列化 ≠ 压缩，但很多“压缩序列化”方案其实就是带压缩的二进制序列化（例如 Protobuf 内部是 varint 编码）。

* #### 文本序列化，是将其`Java`对象变成了字符串之后，然后对其进行编码成字节流然后传输的。

* #### 而二进制序列化，是直接将`Java`对象变成了字节流。中间跳过了一个编码解码的阶段。

  * #### 它是**跳过了字符串阶段**，**直接将对象的字段结构紧凑压缩成字节流**。

  * #### 这种方式**不是对字符串进行压缩**，而是 **对 Java 对象结构做紧凑编码**：字段编号 + 数据类型 + 值。

  * #### 所以说“压缩”是相对 Jackson 来说的数据量更小，是因为**没有走字符串阶段 + 没有字段名 + 数据是紧凑格式**。





# 序列化的目标是谁呢？？——通常是数据体，你所针对的java对象就是这个数据体

## ✅ 二、序列化的目标是谁？（核心问题）

我们来看最典型的 Netty 中「自定义协议结构」：

```diff
+--------+--------+------------+-------------+------------+
| 魔数(4B)| 版本(1B)| 消息类型(1B)| 数据长度(4B) | 数据体(N B) |
+--------+--------+------------+-------------+------------+
```

你要知道：**“谁要被序列化” 是你自己决定的**，通常：

- #### 🧩 魔数、版本、类型、长度：**都是手动写入 ByteBuf 的，没必要序列化**

- #### 🎯 数据体部分：才是你需要“序列化”的 Java 对象（User、Order、RpcRequest等）

```java
// 示例：只对消息体序列化
User user = new User("郑少", 18);
byte[] body = serializer.serialize(user); // ✅ 只序列化 body
```

然后你再手动封装整个数据包结构：

```java
ByteBuf buf = ...
buf.writeInt(MAGIC_NUMBER);
buf.writeByte(VERSION);
buf.writeByte(TYPE);
buf.writeInt(body.length);
buf.writeBytes(body);
```

## ✅ 三、为什么协议头不序列化？

**因为协议头的数据是“结构化、定长、固定字段”，序列化反而麻烦。**

- 你直接 `writeInt()`, `writeByte()` 写入，就完美控制了每一字节在网络包的位置。
- 如果你用 Jackson/Kryo 把整个对象序列化，会多出字段名、顺序无法控制，反而破坏协议结构。

## ✅ 最后总结一句话：

> #### “序列化”不是个魔法，它只负责把**你选中的对象**压成字节或文本，通常只用在“消息体内容”上，**不包含协议头**，协议头是你手动用 ByteBuf 精准拼出来的，**这才是“协议设计”的本质。**

# 我们定义的应用层协议到底是什么？？

## ✅ 一、你说的“协议头”本质是啥？

```text
+----------+--------+--------+---------+---------+
| 魔数(4B) | 版本(1B)| 类型(1B)| 长度(4B) | 数据体(NB) |
+----------+--------+--------+---------+---------+
```

这个数据结构，你是**自定义的应用层协议格式**，和 HTTP、MySQL 协议本质一样：用来**定义两边怎么解析数据的**。

> #### 🔹这些字段（魔数、版本等）其实就是一串 **byte**，不是字符串。
>
>  你可以理解为：

```java
byte[] message = new byte[] {
    0x12, 0x34, 0x56, 0x78, // 魔数
    0x01, // 版本
    0x02, // 消息类型
    0x00, 0x00, 0x00, 0x0A, // 数据长度（10）
    ... // 数据体（10字节）
};
```

也就是说，你说的“字符串”是错的，**网络上传输的是字节数组**，而不是字符串。

## ✅ 二、你这个数据包是属于哪一层？

是的，这是**应用层协议**，属于 TCP 的数据内容部分。

数据在网络上传输是这样的：

```text
[物理层] 电信号
   ↓
[数据链路层] 以太网帧 (MAC 地址、CRC校验)
   ↓
[网络层] IP 报文（源IP、目标IP）
   ↓
[传输层] TCP 包（端口号、序号、ACK等）
   ↓
[应用层] 你定义的协议内容（魔数、版本、消息体等）
```

* #### 所以你的“魔数 + 数据体”这些，**全部都封装在 TCP 的 payload 里**，是最上面那一层。



## ✅ 三、那“序列化”到底是什么？——

我们来个最严谨的定义：

* #### 只是针对应用层协议中的，消息体进行的序列化。而对其他的魔数，或者是其他的东西，没有进行序列化。

* #### 常见的有文本序列化（中间有一层字符串），以及二进制序列化（中间无字符串，直接是java对象转换成二进制数组）

### 📌 序列化（Serialization）：

> #### 是指将 **Java 对象 → 转成字节数组** 的过程，方便网络传输或存储。

比如你有一个对象：

```
User u = new User("张三", 20);
```

你需要把这个对象转成 byte[] 发给对方，那么你必须做“序列化”：

```
byte[] bytes = serialize(u);
```

反过来，对方收到 byte[]，要变成 Java 对象，就要“反序列化”：

```
User u = deserialize(bytes);
```



## ✅ 五、你的终极问题：byte[] → Java 对象 中间发生了什么？

我用你自己的协议流程拆一遍：

1. **Netty 的 ChannelRead 收到数据 → 是 byte[]**

2. 你用 `ByteBuf.readXXX()` 去解析魔数、版本、消息类型、长度

3. 你从 ByteBuf 中读出数据体部分（byte[]）

4. 你根据“消息类型”用对应的反序列化器（比如 Jackson 或 Kryo）：

   ```
   Object obj = serializer.deserialize(dataBodyBytes, User.class);
   ```

5. 你再把这个 `Object` 丢给业务层处理

所以你写的 Decoder 解码器（比如继承 ByteToMessageDecoder），本质就是这个流程的“转换器”：

```
ByteBuf → 魔数校验 → 消息体 → byte[] → JSON解析 → Java对象
```





# 为什么Http这种文本应用层协议解析那么麻烦呢？？

* #### 归根结底。其中间还有一层解码，以及对解码后的字符串进行拆分的过程。虽然有了一定的可读性，不过解析起来非常麻烦。

* #### 因为它是人类早期设计的文本协议，HTTP 请求原始内容就是字符串构成的，比如：

```
GET /index.html HTTP/1.1
Host: www.example.com
Content-Type: application/json
Content-Length: 27

{"name":"zhangsan","age":20}
```

这里你能看到：

- 头部是多行字符串（以 `\r\n` 分隔）
- 空行之后是 body（可能是 JSON、XML、表单）

### 👇所以解析时，流程是这样的：

```
byte[] → 通过 Charset 解码 → String
       → 按照 CRLF 规则切割 → HTTP Header 字典
       → Content-Length 决定 Body 字节读取长度
       → Body 作为字符串再交给 JSON 解析器（如 Jackson）
       → 最终变成 Java 对象
```





# 自定义的应用层协议为什么那么简单的呢？

**是的，没错！自定义应用层协议比 HTTP 简单太多了**，因为它的核心目标是：

> 🚫 不用字符串
>  🚫 不用字符集
>  🚫 不做复杂的多格式兼容
>  ✅ 全部用固定的二进制格式传输
>  ✅ 把一切都定义成 byte[] 的结构解析

## ✅ 对比：HTTP 协议 VS 自定义协议

| 特性     | HTTP 协议                               | 自定义协议                       |
| -------- | --------------------------------------- | -------------------------------- |
| 编码格式 | 字符串（需解码）                        | 二进制（直接 ByteBuf）           |
| 请求解析 | 分割换行符、头部键值、URL解析、参数抽取 | 按字节偏移读取                   |
| 通信效率 | 低（字符串多，体积大）                  | 高（定长字段，compact）          |
| 灵活性   | 高（兼容表单、JSON、XML等）             | 自定义（可定死结构）             |
| 性能开销 | 大（字符集 + JSON 解析）                | 小（直接读 byte[] → 对象）       |
| 适用场景 | 通用浏览器通信                          | 后端服务/游戏/物联网等高性能场景 |

## ✅ 举例：自定义协议解析过程（用 Netty 举例）

定义协议格式：

```
+--------+----------+----------+-----------+-------------+
| 魔数4B | 类型1B   | 序列化类型1B | 数据长度4B | 数据体 N 字节 |
+--------+----------+----------+-----------+-------------+
```

### 服务端解析流程如下：

```java
public class MyDecoder extends ByteToMessageDecoder {
    protected void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) {
        // 1. 魔数校验
        int magic = in.readInt();
        if (magic != 0xCAFEBABE) {
            throw new IllegalStateException("魔数错误");
        }

        // 2. 类型、序列化方式
        byte messageType = in.readByte();
        byte serializerType = in.readByte();

        // 3. 数据体长度
        int length = in.readInt();

        // 4. 数据体读取
        byte[] bytes = new byte[length];
        in.readBytes(bytes);

        // 5. 用自定义序列化器解析
        Object obj = getSerializer(serializerType).deserialize(bytes, getClassByType(messageType));

        out.add(obj);
    }
}
```

### 全流程中没有任何一行是：

- #### 字符串解码（String）

- #### Charset 映射

- #### JSON/String 格式解析

这就带来了**极致的性能和极简逻辑**。

## ✅ 结论：

- 自定义协议就像是你自己在 byte[] 上切分字段，**怎么读都你说了算**
- 不像 HTTP 一样有“行结构”、“头部语法”、“字符集不一致”等一堆恶心事
- 所以 Netty + 自定义协议被广泛用于：RPC、游戏、MQ、Redis 客户端等系统
