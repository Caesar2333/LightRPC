# Websocket到底是什么？和Http的关系是什么？

### ✅ 一句话暴击本质：

> **WebSocket 不是 HTTP 协议，它是一个**“独立的协议”，**但它的握手阶段是**“通过 HTTP 完成的升级请求”。

## 🔧 所以你的几个猜测一一来确认：

------

### ❓**WebSocket 是不是“基于 HTTP 实现”的？**

**❌ 错**。
 WebSocket 的数据传输阶段 **完全不是 HTTP 协议**，它有自己独立的帧格式（frame）和状态管理。

但是：

> ✅ **它的握手阶段确实用的是 HTTP 协议（一次 GET 请求）**，目的是为了穿透防火墙、兼容浏览器、让服务器支持“协议升级”。

------

### ❓**WebSocket 为什么要走一次 HTTP？是不是为了兼容浏览器？**

**✅ 对，完全正确！**

浏览器天然支持 HTTP 请求，但要建立一个双向连接协议，需要先告诉服务器你想干别的。

于是标准制定者搞了个机制：

```http
GET /chat HTTP/1.1
Host: server.example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: xxxxxxx
Sec-WebSocket-Version: 13
```

这是一段 WebSocket 握手请求，其实本质上是：

> **浏览器先发一个特殊的 HTTP 请求，要求升级协议到 WebSocket**。

如果服务器支持，会回复：

```http
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: xxxxxxx
```

返回 101 状态码，表示：我同意你升级协议，从此开始我们不再说 HTTP 的话了。

------

### ❓**升级后还能认为是 HTTP 吗？**

**❌ 不能！升级之后就是完全不同的协议：WebSocket。**

- 数据帧格式不一样（WebSocket 有控制帧、数据帧）
- 没有 HTTP 的 request/response 模型
- 是**全双工**通信，任意一方都能主动发消息
- 持久连接，不走 TCP 连接复用（不是 HTTP/2 的 stream）

## ✅ 那 WebSocket 到底和 HTTP 是啥关系？

| 阶段     | 协议                    | 说明                                  |
| -------- | ----------------------- | ------------------------------------- |
| 握手     | HTTP（GET + Upgrade）   | 借助 HTTP 发起协议升级请求            |
| 成功升级 | 101 Switching Protocols | 服务器同意切换协议                    |
| 建立后   | WebSocket 协议          | 完全脱离 HTTP，自定义帧格式、双向通信 |

## 🧠 补一张脑图级认知：

```
浏览器/客户端
      |
      |   (1) HTTP Upgrade 请求
      V
   ┌──────────────┐
   │   Web 服务器 │ <---- 普通 HTTP 请求还是照处理
   └──────────────┘
           |
           |   (2) 返回 101 Switching Protocols
           V
     WebSocket 双向连接 ←→ 开始收发帧（不再是 HTTP 报文）
```

## 📌 总结一句话你一定记住：

> #### **WebSocket 是用 HTTP“发起”连接的，但握手之后，它就甩掉 HTTP 的壳子，走自己的全双工协议了。**

所以你问的那些：

| 问题                             | 答案                       |
| -------------------------------- | -------------------------- |
| WebSocket 是不是 HTTP？          | ❌ 不是                     |
| 是不是依赖 HTTP？                | ✅ 握手阶段依赖             |
| 是不是为了兼容浏览器设计成这样？ | ✅ 是的                     |
| 握手后本质还是 HTTP 吗？         | ❌ 完全不是，协议层面切换了 |



# 浏览器为啥不能一上来就用 WebSocket 协议？为啥非得先发一个 HTTP 请求，再升级协议？



### ✅ 因为中间那一堆网络设备（如防火墙、代理、网关、CDN、负载均衡器）**它们只认识 HTTP 协议，不认识 WebSocket 协议。**

------

### 🧠 所以 WebSocket 的设计者为了让连接能“穿透这些傻逼设备”，

**才把“握手阶段”做成了一个“合法的 HTTP 请求”，让中间设备放行。**

------

## 🚧 举个最常见的中间网络场景：

假设你访问 `wss://api.example.com/socket`，路径上可能经过：

```
你浏览器
    ↓
家用 Wi-Fi 路由器
    ↓
你 ISP 的 NAT 网关
    ↓
公司防火墙
    ↓
CDN 边缘节点（比如 Cloudflare）
    ↓
HTTPS 负载均衡器（如 Nginx）
    ↓
最终 WebSocket Server
```

### 💥 那问题来了：

这些中间层的设备，比如防火墙/CDN/代理：

- ✅ 它们知道什么是 `HTTP GET /xxx`，也知道怎么处理 `Upgrade: websocket`
- ❌ 但如果你一上来就用自己定义的 WebSocket 协议（不按 HTTP 格式写），它们会直接丢掉，认为你发的是“非法协议”

## ✅ 所以：**HTTP 握手的作用，是“伪装成合法请求让你进门”，升级之后再说“其实我想玩别的”。**

## 🎯 官方标准里就这么规定的

你可以看 RFC 6455（WebSocket 协议标准）里明确说了：

> This protocol **is designed to work over HTTP ports 80 and 443**, and to support intermediaries that might handle HTTP traffic.

意思是：**WebSocket 为了能在现有 HTTP 基础设施中存活**，故意伪装成 HTTP 请求。

## 🚀 那你再来回看这个逻辑：

| 行为                                | 解释                                                         |
| ----------------------------------- | ------------------------------------------------------------ |
| 浏览器不能直接用 WebSocket          | 因为没有独立端口/协议标识，中间设备会阻断                    |
| 为什么用 HTTP GET 携带 Upgrade 头？ | 因为 HTTP 是“合法语义”，中间层会放行                         |
| 握手通过后呢？                      | 客户端和服务端协商成功，直接换成 WebSocket 协议帧通信，中间层不再干涉（如果之前放行了） |

## 🔧 补充：那为啥不定义一个 WebSocket 专属端口？

你可能想问：

> “那 WebSocket 为啥不学 FTP、SSH 直接搞个专属端口 6789 之类？”

答：因为现实就是这样：

- 🧱 公司防火墙/代理一般只开放端口 80（HTTP）和 443（HTTPS）
- 如果你搞个非标端口，**你就会被挡在大厂/企业/校园/CDN之外**
- 所以**WebSocket 设计者干脆走“HTTP通道 + 协议升级”这条路，让你能混进去再说**

## ✅ 最终总结你这轮问题的本质：

| 你问的点                   | 回答                                        |
| -------------------------- | ------------------------------------------- |
| 为啥不能直接用 WebSocket？ | 因为浏览器要“穿透中间设备”，中间层只认 HTTP |
| HTTP 握手是干嘛的？        | 是一段“合法”请求，告诉服务器我要升级协议    |
| 升级后 HTTP 还起作用吗？   | ❌ 完全不再是 HTTP，变成 WebSocket 协议帧    |
| 是为了兼容浏览器吗？       | ✅ 更确切说，是为了兼容“整个 Web 通信链路”   |



#  握手成功之后，WebSocket 还在端口 80 / 443 上跑吗？**要换端口吗？**

> **"This protocol is designed to work over HTTP ports 80 and 443..."**
>  **你问：这是不是意味着用的是 HTTP 的协议？端口是不是也得换？还是说只是“借壳上线”？**

我们一句句翻译、拆透这句话：

## 🧠 一句话直译：

> **这个协议（WebSocket）被设计成可以在 HTTP 的标准端口（80 和 443）上运行，以便穿越可能只允许 HTTP 流量的中间设备。**

## 🔨 那 "work over HTTP ports" 到底是什么意思？

**不是说它是 HTTP 协议，而是说：**

> ✅ 它在使用**HTTP 的端口**（而不是 HTTP 的协议），以便：
>
> - ✅ 浏览器可以正常发出连接
> - ✅ 公司防火墙/CDN/Nginx 不会拦截
> - ✅ 中间代理认为它是“合法 HTTP 流量”，先让你进门

⚠️ 注意：“work over” 这里的意思是“**运行在...之上**（即端口复用）”，不是“本质是 HTTP”。

## 🔥 那你问得最关键的一点来了：

> ❓ 握手成功之后，WebSocket 还在端口 80 / 443 上跑吗？**要换端口吗？**

### ✅ 答：**不换！继续用 80 或 443！**

**协议变了（从 HTTP → WebSocket），但连接没有断，也没切换端口。**

具体来说：

| 阶段     | 协议                       | 端口                         | 说明                                  |
| -------- | -------------------------- | ---------------------------- | ------------------------------------- |
| 握手     | HTTP 请求（GET + Upgrade） | 80（ws://） or 443（wss://） | 合法 HTTP 请求，能穿透                |
| 成功升级 | WebSocket 协议（帧）       | 继续使用原 TCP 连接          | 协议变了，但连接不变                  |
| 后续通信 | WebSocket 自定义帧         | 还是同一个端口               | 不再是 HTTP，不走 request/response 了 |

## 📌 所以你可以记住一句话：

> **WebSocket 是“借用 HTTP 的端口开门”，但“门一开就换成自己说话方式”了，连接和端口没动，只换了协议格式。**

------

## ✅ 你问得太对了：**不是“协议换了就要换端口”，而是“端口不变，只是协议切换”**

举个例子你会秒懂：

### 浏览器连 WebSocket：

```
wss://api.example.com/socket
```

其实它是：

- 向 `api.example.com:443` 发起一个 HTTPS（TLS）连接
- 然后在这个 TLS 流上发出 HTTP 升级请求
- 成功后：继续用这个 TLS 通道发送 WebSocket 的数据帧（不是 HTTP 报文了）

所以：

> ✅ WebSocket 本质上就是 TCP（+ TLS），只是**起手式是 HTTP**罢了。



# 「WebSocket 和 HTTP 是什么关系？WebSocket 是基于 HTTP 吗？」（面试版本）

### 🧠 回答模板：

> WebSocket 并不是基于 HTTP 的协议，它是一个**完全独立的双工通信协议**。
>  不过，为了兼容浏览器和现有的网络基础设施（比如防火墙、代理、CDN），WebSocket 的 **连接建立阶段** 是通过一次 **标准的 HTTP 请求来完成握手升级** 的。

具体来说：

1. 客户端（通常是浏览器）先发起一个带有 `Upgrade: websocket` 的 HTTP 请求。
2. 服务端若支持 WebSocket，会返回一个 `101 Switching Protocols` 响应。
3. 握手完成后，客户端和服务端会在**原有的 TCP 连接上切换协议**，开始使用 WebSocket 协议进行通信，数据以帧（frame）形式双向传输。

所以：

- **握手阶段使用的是 HTTP 协议**（为了穿透中间网络设备）
- **数据传输阶段是完全独立的 WebSocket 协议**（不再是 HTTP 报文）
- **端口通常仍是 HTTP 的 80 或 HTTPS 的 443**，但这只是为了兼容网络，不代表协议还是 HTTP

------

### 📌 总结句（记住这句收尾）：

> **WebSocket 是“借用 HTTP 打开门”，但“一旦进入，就完全换了一套说话方式”，本质是独立的双向通信协议。**
