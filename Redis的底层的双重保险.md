# Redis的IO模型是什么？？？

## ✅ 结论先行：

> **Redis 底层没有使用 Netty**，而是它自己实现了一个高性能的 **事件驱动的多路复用 I/O 模型**，底层用的是 `epoll`、`select`、`kqueue` 等系统调用（视操作系统而定）。**是非阻塞 + I/O 多路复用模型**。

## 🔍 一、Redis 底层是否使用 Netty？

- ❌ **没有使用 Netty**
  - Redis 是用 **C 语言手写的**，追求极致性能，自己实现了网络通信模型（通过 `ae_event_loop`）。
  - Netty 是 Java 编写的通用网络通信框架，用于构建高性能的网络服务器，如 Dubbo、gRPC 的底层等。

所以你不能说“Redis 用 Netty”，它连 JVM 都不需要，纯 C。

## 🔍 二、Redis 的 I/O 模型是啥？

Redis 用的是：

> ✅ **非阻塞 I/O（non-blocking I/O）+ I/O 多路复用（I/O multiplexing）**

### ✅ 什么是“非阻塞 + 多路复用”？

我们用一句面试版的话来总结：

> Redis 使用 epoll 等内核机制对多个 socket 的 I/O 事件进行监听，采用非阻塞方式处理所有客户端连接，避免了为每个连接创建线程，从而实现了高并发低延迟的性能。

也就是：

- **非阻塞**：单个 socket 的 read/write 调用是非阻塞的，数据没到立刻返回；
- **多路复用**：通过 `epoll/select/kqueue` 同时监听大量 socket，有哪个就绪就处理哪个。

### ✅ 多路复用器实现（源码角度）：

Redis 抽象了一个统一的事件处理器：

```
aeApiPoll() -> 实际调用 epoll_wait() / select()
```

不同平台使用不同的后端：

| 平台      | 后端 I/O 多路复用机制 |
| --------- | --------------------- |
| Linux     | epoll                 |
| macOS     | kqueue                |
| Unix 通用 | select                |

## 🔬 三、Redis 为什么不用线程池？为啥这么牛能抗住高并发？

> 因为 Redis 走的是 **单线程 + 多路复用 + 非阻塞 IO** 的极致优化路线。

- 它只需要一个线程处理网络 I/O；
- 通过事件循环（Event Loop）处理所有客户端请求；
- **CPU 和内存都用在了处理命令和数据结构上，而不是线程切换或锁竞争上。**

Redis 作者一句话核心哲学：

> “我宁愿 CPU 忙着执行逻辑，而不是上下文切换。”

## 🚀 所以面试怎么说？

> Redis 没有使用 Netty，它是用 C 写的，自己实现了一个事件驱动的 I/O 框架，底层基于 epoll（Linux）或 select（通用）做 I/O 多路复用，是经典的单线程非阻塞 I/O 设计，核心是用最少的资源处理最多的连接。





# 非阻塞 I/O 和多路复用 I/O 难道不矛盾吗？？

> **非阻塞 I/O 和多路复用 I/O 不仅不矛盾，而且必须共存，组合在一起才能真正发挥威力。**

它们解决的是**不同层级的问题**，属于**不同维度的优化策略**。

## 🔍 先来拆解这两个词到底是什么：

### 1️⃣ 非阻塞 I/O（non-blocking I/O）

- 是针对**单个文件描述符（socket）**的操作行为。
- 比如：`read(fd, buf, len)`
  - 如果没有数据，立刻返回 `-1` 并设置 `errno = EAGAIN`，**不会阻塞等待数据来**。
- 也就是说：**用户态线程自己尝试访问内核 socket 缓冲区的数据，失败了也不阻塞。**

➡️ 非阻塞 I/O 是针对系统调用的行为：**你主动去试探数据来没来**。

------

### 2️⃣ I/O 多路复用（select/poll/epoll）

- 是**一种机制**：我不主动每个都试探，而是问内核：**你告诉我哪个 fd 就绪了，我再去操作**。
- 通过内核提供的系统调用（如 `epoll_wait()`），把多个 fd 注册进去，**只等有事件的 fd 返回**。

➡️ 多路复用的本质是：**让一个线程高效管理多个 socket 的就绪事件**。

------

## 🔄 它们是如何协同工作的？

> 🔥 多路复用 + 非阻塞，正是高性能服务器的主流组合方案（如 Redis、Nginx、Netty、Tomcat（NIO 模式））。

### 📌 原则是这样的：

1. 所有 socket 都设置成 `O_NONBLOCK`（非阻塞模式）；
2. 将这些 socket 注册到 epoll（或 select）中；
3. 等待事件到来（比如有数据可读） → `epoll_wait()` 返回就绪的 fd；
4. 然后你再调用 `read(fd)` 去读；
   - 这时候数据通常已经到了，可以直接读；
   - 如果数据量不够，`read()` 会立刻返回 EAGAIN，你就知道要等下次再来。

## 👀 你提到的误解分析：

> “非阻塞是用户态自己去试探，I/O 多路复用是内核帮你轮询，不是冲突了吗？”

✔ **并不冲突，反而是协作的两部分**：

| 模块          | 做什么                                                     | 举例                                      |
| ------------- | ---------------------------------------------------------- | ----------------------------------------- |
| 内核（epoll） | 监视哪些 socket 上有事件发生，通知用户态                   | `epoll_wait()`                            |
| 用户态线程    | 对通知的 socket 做实际读写（read/write），并且非阻塞地调用 | `read(fd)`、`write(fd)` 配合 `O_NONBLOCK` |

你要记住：**epoll 并不帮你读取 socket，只是告诉你“可以读了”**，你还是要自己 read，但这个 read 是非阻塞的。

## 🚀 面试回答模版：

> 多路复用和非阻塞不是矛盾的，而是协同工作的：
>
> - 非阻塞 I/O 是设置 socket 的 read/write 行为不阻塞；
> - 多路复用是通过 epoll/select 等系统调用，集中等待多个 socket 上的事件；
> - 常见的高性能网络框架（如 Redis、Netty、Nginx）都会结合使用：**用 epoll 管事件，用非阻塞 I/O 实现高效读写**，避免线程被挂起，提高并发能力。



# Redis的双重保险

>明白了，换句话说 其做了双重保险对吧？？ 这里的非堵塞io和我们之前说的 广义上的非堵塞io不一样的。 之前的非堵塞io是你用户态自己去轮询 所有的socket的，而redis的非堵塞io是 你的epoll通知你了，告诉你了，这里可以读，不过不一定有，单个的socket的 非堵塞io对吧？？

## 🎯 是的，**这就是双重保险 + 分工明确的优化机制**：

我们说的 Redis 模型，其实是下面这两层 **同时存在**：

| 层级                           | 技术机制                  | 解决的问题                                                   | 举例说明                                             |
| ------------------------------ | ------------------------- | ------------------------------------------------------------ | ---------------------------------------------------- |
| **宏观层面（多 socket 管理）** | `epoll` 多路复用          | 用一个线程监听成千上万个 socket，谁准备好了就告诉你，不必自己一一 poll | “告诉我哪些 socket 有事，不用我亲自挨个试”           |
| **微观层面（单 socket 行为）** | 非阻塞 I/O (`O_NONBLOCK`) | 即使你告诉我准备好了，我 read 的时候也不能阻塞               | “就算你说好了，我 read 一下还是不等你，有就读没就走” |

## 🧠 再来换种说法，更通俗：

你可以想象：

> epoll 是“门口的保安”，它说：嘿，13号 socket 有人敲门了，你去看看。
>
> 然后你就过去试着 `read(13号)`，但门一开，发现人还没来（数据还没准备好），你不会傻站着等 —— 因为你是**非阻塞模式**，你说：“那我晚点再来”，然后继续干别的事。

所以 Redis 做的就是：

- 用 epoll 快速过滤掉“有动静”的 socket；
- 每个 socket 都是非阻塞的 → 即使你骗我“准备好了”，我也不被你卡死。

### ✅ 正是这 **组合拳** 才让一个线程顶千军万马：

- 减少了无意义的系统调用（不自己轮询全部 socket）；
- 减少了线程挂起、上下文切换（非阻塞）；
- CPU 时间都花在了逻辑处理和数据搬运上，性能极高。

## 🔍 你提到的重点：**“广义上的非阻塞 I/O” vs “Redis 中的非阻塞 I/O”**

精准！

- 🔁 **传统意义上的非阻塞 I/O（non-blocking I/O）**：
  - 是“主动式”：用户线程主动 `read(fd)`，一个个试。
  - 问题：你还是得**遍历所有 socket**，对高并发不友好。
- ⚙️ **Redis 中非阻塞 I/O 的用法**：
  - 是在 epoll 提供的“有动静的 fd 列表”中，配合非阻塞 socket 处理具体的 I/O。
  - 所以 Redis **并不自己轮询全部 socket**，而是“你告诉我哪个可以读，我来试试（但我也不等）”。

## ✅ 总结成一句话（面试 / 笔记可用）：

> Redis 网络通信使用了 **epoll + 非阻塞 I/O** 的组合，其中 epoll 是事件级别的多路复用，用于监听成千上万的 socket 是否就绪，而每个 socket 又设置为非阻塞，保证即使被通知也不会卡死线程。这是一种 **“协作式的非阻塞模型”，而不是传统意义上用户线程轮询全部 socket 的死板非阻塞操作**。
